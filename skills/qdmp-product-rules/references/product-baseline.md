# 千岛小程序产品底线原则

本文基于千岛小程序产品底线原则整理，仅抽取代码生成、脚手架、组件约束、代码 Review 能明确执行的部分；偏产品判断、业务取舍、内容运营判断的原则不放入代码生成 Skill，避免模型用模糊标准替代产品/审核判断。

## 1. 适用范围

这份规则用于约束千岛小程序的代码生成与改造，包括：

- 新建小程序项目、页面、路由、组件、状态管理、接口调用；
- 生成页面布局、交互流程、权限申请、登录流程；
- 引入千岛能力，例如 SPU、识图、估价、商品/帖子跳转、唤起 App 原生发帖；
- 代码 Review 时识别明显越界的产品形态。

这份规则不用于替代产品审核。凡是涉及「是否允许」「是否需要特批」「千岛承载不了但场景确实需要」的判断，应回到开放平台产品审核侧确认。

## 2. 代码生成 Skill 可以强约束的底线

### 2.1 小程序必须是单一工具，不生成 App 化结构

生成规则：

- 每个小程序默认只围绕一个核心任务生成代码。
- 页面路由应围绕一条主流程组织：进入 → 操作 → 得到结果 → 离开。
- 默认不生成底部 TabBar、多 Tab 导航或「首页 / 发现 / 消息 / 我的」式结构。
- 页面层级尽量浅；能在一个页面完成的流程，不主动拆成多个无必要页面。

禁止生成：

- tabBar 配置；
- Home / Discover / Message / Mine 等 App 化页面骨架；
- 多个互不相关的一级功能入口；
- 以「平台」「社区」「个人中心」为主的导航结构。

Skill 检查点：

- 如果需求要求底部 Tab、多频道、多栏目，默认拒绝按 App 形态生成，并建议改成单流程工具形态。

### 2.2 不重复建设千岛主 App 的社区核心能力

生成规则：

- 涉及内容沉淀、发帖、展示社区内容时，优先使用千岛已有能力或唤起千岛 App 原生能力。
- 小程序产生的结果内容，例如战报、估价结果、组队卡片，应优先设计为「生成结果 → 唤起千岛原生发帖 / 跳转千岛内容」。

禁止生成：

- 自建发布/动态系统；
- 帖子 Feed、内容流、推荐流；
- 关注/粉丝/私信；
- 小程序内的个人主页、我的动态、社区互动闭环；
- 在小程序内保存并消费一套独立社区内容。

Skill 检查点：

- 遇到「发动态、帖子流、关注、评论、私信、个人页」等需求，不能直接生成自建实现。
- 需要内容回流时，生成「唤起千岛原生发帖」或「跳转千岛内容」的技术方案。

### 2.3 不自建账号体系，按需使用千岛账号与权限

生成规则：

- 登录态、用户身份、授权能力优先接入千岛现有能力。

禁止生成：

- 独立用户身份体系。

Skill 检查点：

- 默认使用千岛的 login 作为用户登录。

### 2.4 不做留存、激励、召回型机制

生成规则：

- 小程序完成任务后，应提供结果、分享/回流千岛能力或退出路径，不主动制造留存闭环。
- 代码生成中不要主动添加运营留存模块。

禁止生成：

- 签到；
- 积分；
- 会员等级；
- 每日任务；
- 召回 Push；
- 为延长停留时长而设计的推荐流、活动流、任务流。

Skill 检查点：

- 如果需求包含签到、积分、等级、每日任务、Push 召回，不能直接生成；应提示这类能力属于产品底线风险，需要产品确认。

### 2.5 不自建独立资料库，优先写入/引用千岛资料库

生成规则：

- SPU、商品、价格、识图结果、资料类信息应优先使用千岛资料库或写入对应 SPU。
- 小程序本地或业务侧只保存完成当前工具任务所需的临时状态、缓存或必要业务记录。

禁止生成：

- 与千岛资料库平行的一套商品库、资料库、价格库；
- 只在小程序内生效、无法回流千岛 SPU 的长期资料沉淀；
- 用本地静态数据或独立后端长期替代千岛资料库。

Skill 检查点：

- 资料沉淀类需求默认要求接入千岛资料库或明确 SPU 归属。

### 2.6 页面上不出现不可用能力或占位功能

生成规则：

- 页面中展示的按钮、入口、菜单项必须有可执行逻辑和明确结果。
- 未实现能力不生成 UI 入口。
- 灰态按钮只用于短期运行态，例如加载中、提交中、权限校验中，不用于长期占位。

禁止生成：

- 「敬请期待」；
- 「即将上线」；
- 长期不可点击的灰按钮；
- 没有 handler 的按钮；
- 空实现、TODO 入口、假数据入口。

Skill 检查点：

- 生成页面后检查所有可见入口是否绑定真实动作。
- 如果能力未实现，直接从 UI 中移除，而不是生成占位。

### 2.7 核心任务用最短路径完成

生成规则：

- 优先使用识图、估价、SPU、商品/帖子跳转等千岛能力减少手动输入。
- 能选择就不手填；能自动带出就不让用户重复填写。
- 表单字段只保留完成当前任务必要的信息。
- 默认不生成欢迎页、新手引导页、冗长确认页。

禁止生成：

- 与当前任务无关的表单项；
- 可由千岛能力自动获得、却要求用户手动填写的信息；
- 进入小程序后的纯欢迎页；
- 非必要的新手引导；
- 多余的二次确认和跳转步骤。

Skill 检查点：

- 生成流程前先列出用户完成任务的最短步骤。
- 如果步骤中存在可由千岛能力替代的输入，改为调用能力或选择器。

## 3. 不建议放入代码生成 Skill 强约束的内容

以下内容可以保留在产品原则或人工 Review 中，但不适合作为代码生成 Skill 的硬规则：

- 「一个小程序只解决一个问题」中的具体业务边界判断：模型可以提醒收敛，但不能替产品判断某个场景是否算一个问题。
- 「一个页面只有一个重点」中的视觉重心、信息优先级判断：可以转化为减少无关模块，但不应让模型主观判断页面是否“足够聚焦”。
- 「好的工具靠价值让用户回来」这类价值判断：不进入 Skill。
- 「千岛承载不了但场景确实需要」的例外判断，例如组队房间、组队历史：模型只能标记为黄灯，不能自行放行或拒绝。
- 设计稿美观度、运营策略、业务优先级、审核尺度等非代码可验证事项。

## 4. 代码生成 Skill 的执行模板

### 4.1 生成前判断

在生成千岛小程序代码前，先检查需求是否触发以下风险词：

- App 化结构：底部 Tab、首页、发现、消息、我的、多频道、多栏目；
- 社区能力：发动态、帖子流、Feed、关注、粉丝、私信、评论、个人页；
- 留存机制：签到、积分、等级、每日任务、Push 召回、活动中心；
- 自建体系：自建账号、自建资料库、自建商品库、自建内容库；
- 占位功能：敬请期待、即将上线、灰按钮、TODO 入口。

命中后不要直接生成，应先改写为符合底线的技术方案，或提示需要产品确认。

### 4.2 生成时默认策略

- 默认单任务、单主流程、浅层页面。
- 默认不生成 TabBar。
- 默认不生成社区闭环。
- 默认不生成自建账号体系。
- 默认不生成长期留存/激励/召回模块。
- 默认不生成不可用入口。
- 默认按需登录、按需授权。
- 默认优先调用千岛能力，减少用户手填。

### 4.3 生成后 Review 清单

提交代码前检查：

- 是否存在 tabBar 或 App 化页面结构？
- 是否存在自建发布、Feed、关注、私信、个人页？
- 是否存在签到、积分、会员等级、每日任务、Push 召回？
- 是否存在自建登录/注册体系？
- 是否存在独立资料库、商品库、价格库？
- 是否存在「敬请期待」「即将上线」或无 handler 的按钮？
- 是否有可由千岛能力自动带出的字段仍要求手填？
- 小程序结果是否能回流千岛或跳转千岛能力，而不是只留在小程序内？

## 5. 黄灯场景处理

当需求确实需要小程序内承载某些千岛主 App 不能直接承载的形态时，例如组队房间、组队历史等，代码生成 Skill 不应自行判断通过。

处理方式：

1. 标记为黄灯场景；
2. 暂停生成最终实现；
3. 输出需要产品确认的问题：
   - 这个形态是否允许在小程序内承载？
   - 数据是否需要回流千岛？
   - 是否需要限制展示范围和生命周期？
   - 是否有审核口径或开放平台约束？
4. 产品确认后，再按确认结果生成技术方案。
