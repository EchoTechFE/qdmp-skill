# PRD 模版说明

## 什么是好的 PRD

一份好的 PRD 应该让产品、开发、设计、测试都能从中找到自己需要的信息，而不需要反复追问。核心标准：

- **背景清晰**：为什么做这个，解决谁的什么问题
- **边界明确**：做什么、不做什么，都要说清楚
- **流程完整**：用户怎么一步步完成目标，异常情况怎么处理
- **状态可追踪**：每类数据有哪些状态，怎么流转
- **验收可测**：每个功能点都能写出测试用例

## 一、背景与目标

### 背景
[为什么要做这个产品/功能？现在有什么问题或机会？]

示例：用户在社区内有大量闲置物品，目前只能通过私信交易，流程繁琐且缺乏保障。

### 目标
[做完之后，期望达到什么效果？尽量可量化]

示例：
- 用户可以在小程序内完成从发布到交易的完整流程
- 上线后 30 天内 GMV 达到 10 万元

### 不在范围内
[明确说明哪些不做，避免范围蔓延]

示例：
- 不支持多件商品合并下单
- 不做物流跟踪，仅记录快递单号

## 三、核心功能模块

### 模块 1：[模块名，如"商品发布"]

**功能描述：**
[一句话说明这个模块做什么]

**用户操作流程：**
1. 用户点击「发布」按钮
2. 填写商品标题、价格、图片（最多9张）、描述、分类
3. 点击「提交」，商品进入审核状态
4. 审核通过后，商品自动上架展示

**异常情况：**
- 图片上传失败：提示重试，不影响其他字段
- 审核不通过：通知用户原因，可修改后重新提交

**涉及的数据：**
- 商品：标题、价格、图片、描述、分类、状态、发布者、发布时间

**商品状态流转：**
```
待审核 → 在售（审核通过）
待审核 → 审核不通过（可重新提交）
在售 → 已售出（买家付款后）
在售 → 已下架（卖家手动下架）
```

**查询场景：**
- 我的发布：查看当前用户发布的所有商品（含各状态）
- 商品详情：查看某件商品的完整信息

**验收标准：**
- [ ] 用户可以上传 1-9 张图片
- [ ] 价格只能输入正数，最多两位小数
- [ ] 提交后展示"审核中"状态
- [ ] 审核通过后商品出现在列表页

## 四、数据关系

[按实体列出它与其他数据的关联。模块较少时用表格，模块较多（5个以上）时推荐用 Mermaid ER 图。]

**表格方式（模块较少时）：**

| 实体 | 关联实体 | 说明 |
|------|----------|------|
| 用户 | 商品 | 一个用户可以发布多件商品，一件商品只属于一个用户 |
| 用户 | 订单 | 一个用户可以有多个买家订单 |
| 商品 | 订单 | 一件商品只能被购买一次（售出后自动下架） |
| 商品 | 评论 | 一件商品可以有多条评论 |
| 用户 | 评论 | 一个用户可以发表多条评论，评论属于某件商品 |

**树形结构方式（模块较多时）：**

```
用户
├── 发布 → 商品（多件）
│         └── 包含 → 评论（多条）
├── 购买 → 订单（多个）
│         └── 对应 → 商品（一件）
└── 发表 → 评论（多条）
```
